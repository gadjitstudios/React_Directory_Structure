Introduction
React is an very unopinionated language, which is great because it allows the developer(s) to create a structure/style of coding that meets the needs of the project as opposed to a list of rules thought up by the language's creators. There are some downsides to this, especially when you are beginning to learn the language. One of the issues I had with my first React project was with the layout of my project's directory structure. There are some resources out there to help you figure out somewhat of a basic structure; such as this article from React's website:  https://reactjs.org/docs/faq-structure.html; but, if you dig any deeper you are likely to find some conflicting opinions. 
There is one other important concept we must talk about before we get into the thick of it, and that is the idea of single responsibility. Basically what this states is that a module must only be responsible for doing one thing. We will add to this discussion on single responsibility when we start talking about the individual directories; and what better time to do that then now. So, having said that, I thought that I would throw my hat into the ring, and give you my opinion on a directory structure. Here it is:

Components
I like to think of components as the "micro" level of React. These are small, reusable components such a custom buttons, picture galleries, etc. The best way to figure out whether you're module should go in this directory is to ask, "will I be able to rip this code out of this project and place it in a different one, and NOT require any other modules (3rd party, such as Redux or Bootstrap, excluded)?" If the answer is yes, you are allowed to add your module to the components directory. There are two more rules of the components directory that you must follow: 
  • each component must have its own directory inside of the components directory
  • each component directory inside of the components directory must contain only that components source files
Views
Views are made up of a collection of components from the component directory, and/or individual html elements. A view's single responsibility is to display elements or components to the user. In doing this, a view will collect data from other modules and use that data to produce the view. Remember, view only gathers this information, it does nothing to produce it. 

Services
A service is a module that is used to provide, validate, or manipulate data for use in another module. The most common service I create in my React apps is an API service, who's single responsibility is to make REST calls to servers (typically the app's backend). I have, in the past, also created a service to validate form data using Regex checks. These are just some examples of what you might find in the services directory.

Assets
This is a pretty common naming convention/ style. Typically the Assets directory contains image, logos, and other resources used throughout the app. The Assets directory should contain other directories that will organize these resources accordingly. Please do yourself a favor and do not just pile all of your resources directly in the Assets directory. Also, remember that a view is responsible for collecting an assembling what the user sees, so it is most likely that a majority of your resources will be imported in a view an passed to other components.

Stores
This directory is optional, and should only contain state store modules, such as a Redux store, a custom "use" hook, etc. Basically, if it acts like a store, it goes here.

Common File Types
Now that we have our directories defined, we need to talk a little bit about the files you will find inside of a directory, specifically the views and components directories. There are 3 common file types that these directories should contain:
  1. a React file (js or jsx)
  2. a style file (css, sass, scss)
  3. a data file (js, json, etc)
The first 2 are pretty self-explainitory, React files create React components (an import the other 2 file types); and, the style file adds css to the React component. The last file, a data file, is one that we have not really explored yet. This file's single responsibility is to collect data and supply that data to the React component. This file should also handle cleaning and storing that data.
